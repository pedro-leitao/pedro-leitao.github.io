Getting Size Right: How Engineering Teams Scale

larger doesn’t always mean faster or better
Thoughts
Software Engineering
Scaling
Published

March 4, 2025

I have spent a significant portion of my career trying to understand how engineering teams work together and scale, and working out ways to assist them in becoming more effective. If you have been in the same position, you probably know that it is a job which is never truly finished, and which usually leads to no gratifying conclusion or a “thank you” at the end.

Working with people is hard, understanding personalities, how different people collaborate, and how to help them work better together is extremely difficult.

Despite all this, there is often a push to grow teams when something isn’t quite working or objectives are not met. The underlying assumption is that more people will get the job done faster, and as importantly, that people are fungible and can just be moved around and hit the ground running.

This couldn’t be further from the truth, and it usually leads to utter and complete disaster. Non-performing teams most of the time are so for a number of reasons, and it is uncommon for size to be the main one.

from bokeh.plotting import figure, show
from bokeh.layouts import column
from bokeh.models import Slider, ColumnDataSource, CustomJS
from bokeh.io import output_file
import numpy as np

# Initial Slider Values
comm_init = 0.1  # Initial communication percentage
integ_init = 0.1  # Initial integration percentage
team_size_init = 7  # Initial team size

# Constant for the model
a = 0.25

# Compute initial values
n = team_size_init
T_c_init = comm_init * (1 - np.exp(-a * n))
T_i_init = integ_init * (1 - np.exp(-a * n))
P_init = max(0, 1 - T_c_init - T_i_init)

# Data source for the bar chart
source = ColumnDataSource(
    data=dict(
        categories=["Productivity", "Communication", "Integration"],
        values=[P_init, T_c_init, T_i_init],
        colors=["#1f77b4", "#ff7f0e", "#2ca02c"],
    )
)

# Create the plot
plot = figure(
    title="Team Productivity Breakdown",
    x_range=["Productivity", "Communication", "Integration"],
    y_range=(0, 1),
    x_axis_label="Category",
    y_axis_label="Fraction of Time",
    width=600,
    height=400,
    tools="",
)
plot.vbar(
    x="categories",
    top="values",
    width=0.9,
    source=source,
    fill_color="colors",
    line_color="white",
)

# Add sliders
communication_slider = Slider(
    start=0, end=1, value=comm_init, step=0.01, title="Communication (%)"
)
integration_slider = Slider(
    start=0, end=1, value=integ_init, step=0.01, title="Integration (%)"
)
team_size_slider = Slider(
    start=2, end=10, value=team_size_init, step=1, title="Team Size"
)

# JavaScript callback to update the plot
callback = CustomJS(
    args=dict(
        source=source,
        comm_slider=communication_slider,
        integ_slider=integration_slider,
        team_slider=team_size_slider,
    ),
    code="""
    const c = comm_slider.value;
    const i = integ_slider.value;
    const n = team_slider.value;
    const a = 0.25;
    const T_c = c * (1 - Math.exp(-a * n));
    const T_i = i * (1 - Math.exp(-a * n));
    const P = Math.max(0, 1 - T_c - T_i);
    source.data.values = [P, T_c, T_i];
    source.change.emit();
    """,
)

# Attach the callback to the sliders
communication_slider.js_on_change("value", callback)
integration_slider.js_on_change("value", callback)
team_size_slider.js_on_change("value", callback)

# Combine sliders and plot into a layout
layout = column(communication_slider, integration_slider, team_size_slider, plot)

# Save and show the plot
output_file("team_productivity_breakdown.html")
show(layout)
